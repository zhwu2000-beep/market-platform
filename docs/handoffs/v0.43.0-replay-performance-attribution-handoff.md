## E. Minimal Production Change

`HistoricalReplayService.run()` remains the only production historical replay loop.

The only production code adjustment is a private helper extraction:

```python
def _copy_replay_prefix(prices: pd.DataFrame, position: int) -> pd.DataFrame:
    return prices.iloc[: position + 1].copy(deep=True)
```

This helper exists only to create a real timing boundary for prefix slicing/copy. It does not change public API, deep-copy semantics, full-frame position behavior, input immutability, replay models, serialization, or point-in-time behavior. The helper is private and is not exported.

The benchmark harness does not copy the replay loop.

## F. Benchmark Architecture

The benchmark uses two tracks.

Uninstrumented production timing:

- directly calls `HistoricalReplayService.run()`
- uses no patching, wrappers, instrumentation, or monkeypatching
- is the authoritative wall-clock production timing

Instrumented attribution timing:

- still calls the real `HistoricalReplayService.run()`
- patches only the replay module bindings that the production service actually uses
- wraps injected services/models with transparent call-through proxies
- records segment durations and call counts without changing return values, inputs, order, or exceptions

Benchmark controls:

- DataFrame construction is outside timing.
- The default seed is fixed at `43`.
- The harness generates one deterministic max-bars frame, then gives 100/300/500 scenarios deep-copied prefixes of the same frame.
- Warm-up runs and measured runs are separate.
- `gc.collect()` runs before every warm-up and measured timing.
- Serialization is measured only after `HistoricalReplayService.run()` returns.
- No network or provider API is used.

## G. Measured Segments

Instrumented attribution records:

- price normalization
- provider extraction
- replay-position selection
- signal precompute
- strategy identity construction
- prefix slicing/copy
- `PriceStructureService.analyze()`
- `build_historical_market_observation()`
- `state_model.evaluate()`
- `StrategyRunner.evaluate()`
- orchestration/model/unattributed residual

Segment timing comes from instrumented samples. Production total comes from uninstrumented samples. These are independent runs. Instrumentation overhead and sample noise mean segment shares are attribution ratios and are not guaranteed to sum exactly to 100%.

The 300-bar overhead ratio in the formal run was `0.94x`; this is independent sample noise and does not mean instrumentation speeds up production replay.

## I. v0.42.1 Comparability

The v0.42.1 handoff retained older benchmark numbers, but the repository does not retain the corresponding benchmark script, data generator, or complete command.

Because the v0.42.1 and v0.43.0 workloads cannot be proven identical, their absolute timings are not directly comparable. The current v0.43.0 benchmark workload is deterministic, oscillatory, and structure-heavy. It creates substantial pivot/candidate/zone/touch work. The older v0.42.1 benchmark may have used simpler monotonic or near-monotonic data.

Do not use the difference between those absolute timings to claim a performance regression.

## J. Decision

Decision: GO - candidate `v0.44.0 Price Structure Replay Precompute`.

Basis for this benchmark workload:

- Structure analysis is the largest measured segment.
- At 500 bars, structure analysis is about `89.2%` of production median as a normalized attribution ratio.
- Structure workload grows strongly with bars.
- Signal precompute, prefix copy, state evaluation, and strategy evaluation are not the priority bottlenecks under this workload.

Limits:

- This does not prove every real market-data workload has the same segment distribution.
- v0.44.0 cannot eliminate all replay time.
- Observation construction is likely to become the next bottleneck after structure optimization.
- Instrumented segment totals include wrapper overhead and independent sample error.

## K. v0.44.0 Goals

Candidate next stage: `v0.44.0 Price Structure Replay Precompute`.

Target behavior:

- Precompute point-in-time `PriceStructureSnapshot` values once over the full normalized replay frame.
- Make each replay step consume `structure_snapshots[position]` by full-frame position.
- Avoid per-prefix `PriceStructureService.analyze(prefix)` calls during replay.
- Keep public `PriceStructureService.analyze()` API unchanged.
- Keep structure models, schema, and serialization unchanged.
- Keep the structure package independent of replay.
- Preserve full-frame positions when replay uses `start`/`end` subwindows.

## L. v0.44.0 Required Parity

For every full-frame position `i`, precomputed structure must be strictly equivalent to:

```python
PriceStructureService().analyze(
    normalized.iloc[: i + 1],
    as_of=normalized.iloc[i]["timestamp"],
)
```

Parity must cover:

- availability/status
- `as_of`
- pivots
- candidate levels
- clustered levels/zones
- support/resistance zones
- touch observations
- ATR
- parameters/configuration
- provenance/source fields
- ordering
- `to_dict()` and JSON serialization
- warm-up boundaries
- future-data isolation
- unsorted input behavior
- input immutability

## M. v0.44.0 Design Risks

Known risks to resolve before implementation:

- pivot confirmation may depend on future bars if implemented carelessly
- ATR warm-up boundaries
- point-in-time level clustering state
- stable zone identity and ordering
- touch observation incremental semantics
- floating-point exact parity
- default fast path versus injected component paths
- private/public service boundaries
- full-frame position mapping
- `start`/`end` subwindow behavior
- model construction cost after structure analysis is optimized
- never fake point-in-time snapshots by copying final structure results backward
