## C. Project Overview

`market-platform` is a Python 3.14 investment research platform built around normalized market data, deterministic analytics, and point-in-time replay. The project script entry point is `market-platform = "market_platform.cli.main:main"`.

Main modules verified in `src/market_platform`:

- `data`: provider interfaces, provider registry/selection, capability checks, provider fallback, normalized DataFrame schemas, health/diagnostics, cache support, and provider implementations for Polygon, Twelve Data, Finnhub, SEC EDGAR, FRED, news, and options surfaces. Current docs state Polygon and Twelve Data are the primary implemented providers for daily/latest/intraday price paths.
- `signals`: pure signal calculators, signal models, composite/classification/ranking helpers, relative strength, batch classification, and the default `calculate_market_signals()` aggregation service.
- `structure`: price structure analysis, swing pivots, ATR, level clustering, zone/touch observation, structure models, and reporting.
- `observation`: immutable market observation models plus builders for live/current and historical point-in-time observations.
- `state`: `MarketState` models, `MarketStateModel` protocol, and `BaselineMarketStateModel` deterministic rule evaluation.
- `strategy`: strategy protocols, strategy collections, configurable instances, validation, synchronous runner, baseline trend strategy, and baseline volatility strategy.
- `replay`: historical replay service, immutable replay result models, and replay summary reporting.
- `research`: end-to-end research workflow, state bridge, interpretation, target framework, price context, reports, strategy adapter, and research models.
- `cli`: commands for market data, provider diagnostics/health, signal classification, research runs, and historical replay.

The architecture keeps providers behind standardized DataFrame contracts. Replay orchestrates lower layers but should not invert dependencies into provider, CLI, or research code.

## E. v0.42.1 Signal Replay Precompute

Original issue:

- Before `v0.42.1`, each replay step built `prefix = normalized.iloc[: position + 1].copy(deep=True)` and called `calculate_market_signals(prefix)`. That repeated signal frame preparation and rolling calculations for every replay step.

New design:

- `src/market_platform/signals/service.py` adds `precompute_market_signal_snapshots(prices)`, intended as an internal service-level batch capability.
- `calculate_market_signals(prices)` public API is unchanged and remains exported from `market_platform.signals`.
- `precompute_market_signal_snapshots()` is not added to `market_platform.signals.__all__`, so the package-level public export was not expanded.
- `signals` does not depend on `replay`; `replay` imports the service-level precompute function.
- `HistoricalReplayService.run()` normalizes the full replay frame, computes `signal_snapshots = precompute_market_signal_snapshots(normalized)` once before the replay loop, and reads `signal_snapshots[position]` per step.
- The replay loop still constructs `prefix` for `PriceStructureService.analyze()` and `build_historical_market_observation()`.

Default signal order and rules:

1. `trend`: trailing SMA20 divided by trailing SMA50 minus 1.
2. `momentum`: `close / close.shift(20) - 1`.
3. `realized_volatility`: 20-period log return standard deviation, `ddof=1`, annualized by `sqrt(252)`.
4. `current_drawdown`: `close / expanding max close - 1`.
5. `distance_from_moving_average`: `close / trailing SMA20 - 1`.

Point-in-time guarantees:

- For position `i`, `precompute_market_signal_snapshots(normalized)[i]` is tested to equal `calculate_market_signals(normalized.iloc[: i + 1])` exactly.
- Warm-up values remain `None` where the prefix calculators would return `None`.
- Future-data isolation is tested by appending future rows and checking that prior snapshots do not change.
- Unsorted input is copied, stably sorted by timestamp, and does not mutate the caller's DataFrame.
- Replay `start`/`end` filtering uses full-frame positions. A run over `start=position 100` reads `signal_snapshots[100]`, not `signal_snapshots[0]` from a sliced subwindow.
- Replay result `to_dict()` parity is tested against a reference replay path that supplies signal snapshots by calling public `calculate_market_signals()` for each prefix.

Realized volatility boundary fix:

- The batch path preserves the old positive-close behavior for realized volatility. For frames longer than the 20-bar volatility warm-up, any `close <= 0` raises `ValueError("Close prices must be positive for realized volatility")`.
- For frames at or below the warm-up threshold, batch realized volatility stays `None`, matching the prefix calculator behavior without computing log returns.

Not included:

- No cache.
- No persistence.
- No multiprocessing.
- No parallel computation.
- No provider, CLI, research, state, or strategy behavior changes.

## G. Architectural Invariants

Do not accidentally break these constraints in the next stage:

- Point-in-time/no-look-ahead behavior is mandatory for signals, structure, observations, state, strategies, and replay summaries.
- Replay `start`/`end` must continue to map to positions in the full normalized frame, not a sliced subwindow.
- Public API compatibility matters: `calculate_market_signals(prices)` and model `to_dict()` schemas must remain stable unless a versioned change explicitly requires otherwise.
- Signal order and strategy evaluation order must remain deterministic.
- Input DataFrames should be copied/normalized without mutating caller-owned frames.
- Provider/CLI/research/state/strategy behavior should not change as a side effect of replay performance work.
- `signals` must not depend on `replay`; replay orchestration must not invert package dependencies.
- Replay package tests currently guard against forbidden dependencies on research, CLI, provider, portfolio, risk, and execution layers.
- Reference parity tests should use public/simple behavior as the oracle, not copy the optimized implementation.
- CLI stdout/stderr/logging contracts should be preserved where tests cover them; inspect CLI tests before changing output or logging behavior.

## H. Known Remaining Costs and Open Questions

Known facts:

- Signal prefix recomputation has been eliminated in `v0.42.1`.
- A 500-bar replay with two default strategies still measured about `7.304s` median locally.
- Replay still performs substantial per-step work.

Potential remaining costs inside or adjacent to the replay loop:

- Prefix DataFrame slicing/copying.
- `PriceStructureService.analyze()`.
- Historical observation construction.
- Observation provenance/fingerprint calculation in `src/market_platform/observation/history.py`.
- `BaselineMarketStateModel.evaluate()`.
- `StrategyRunner.evaluate()` and strategy validation.
- Replay step/result dataclass construction and validation.
- Final `HistoricalReplayResult.to_dict()` serialization when callers request JSON/dict output.

Open questions:

- The repository does not yet contain a reliable segment-level timing attribution for replay.
- Current benchmark numbers do not prove which component dominates after signal precompute.
- Structure analysis may be important, but it should not be assumed to be the largest bottleneck without profiling.

Before another large optimization, establish measured call counts and timing breakdowns.
