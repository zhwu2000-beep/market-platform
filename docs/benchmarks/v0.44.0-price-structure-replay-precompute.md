# v0.44.0 Price Structure Replay Precompute

## Purpose

v0.44.0 implements the first Price Structure Replay Precompute stage for the default historical replay path. The goal was to stop calling `PriceStructureService.analyze(prefix, as_of=...)` once per replay step, while preserving exact point-in-time behavior and replay output parity.

This report records the implementation design, parity method, benchmark workload, final review benchmark, internal structure attribution, and remaining bottlenecks. Git state, code, and tests remain the source of truth.

## Design

Default `HistoricalReplayService.run()` now precomputes structure snapshots once on the full normalized replay frame when the structure service is exactly the default `PriceStructureService` with default component identities.

The replay loop keeps its production shape:

- Price normalization still happens once.
- Signal precompute still happens once.
- Structure precompute now happens once before the replay loop for the default path.
- Prefix deep copies still happen once per replay step for historical observation construction.
- State and strategy evaluation order is unchanged.
- Replay models and serialization are unchanged.

For the default structure path:

- `precompute_price_structure_snapshots(normalized)` returns one `PriceStructureSnapshot` per full-frame position.
- Replay steps read `structure_snapshots[position]`.
- Public `PriceStructureService.analyze()` is not called per step.

For subclassed or injected structure services:

- Replay keeps the prior per-step `service.analyze(prefix, as_of=as_of)` behavior.
- The precompute path does not infer custom component semantics.

## Point-In-Time Pivot Confirmation

Default pivots are detected once on the full prepared frame, but snapshots only include candidates with:

```text
candidate.confirmed_at <= snapshot_as_of
```

This preserves the existing right-window confirmation semantics:

- A pivot occurrence remains invisible before its confirmation bar.
- The pivot becomes visible at the confirmation position.
- Tail candidates without enough right-window bars are never visible.
- Adding future bars does not change earlier snapshots.

The existing pivot model already carries both `occurred_at` and `confirmed_at`, so no schema change was required.

## Exact Parity

The parity contract for every position `i` is:

```python
precomputed[i] == PriceStructureService().analyze(
    normalized.iloc[: i + 1],
    as_of=normalized.iloc[i]["timestamp"],
)
```

Tests compare both dataclass equality and serialized dictionary output through `snapshot_to_dict(...)`. Replay parity tests compare complete `HistoricalReplayResult.to_dict()` output between the optimized default path and an injected reference service that calls the public prefix analyzer per step.

The implementation shares normalized snapshot assembly logic between public `analyze()` and precompute to avoid maintaining separate status/model construction paths.

## Benchmark Workload

The benchmark preserves the v0.43.0 workload:

- Deterministic synthetic data generator
- Default seed: `43`
- Single max-bars frame generated once
- 100 / 300 / 500 bar scenarios use deep-copied prefixes from the same max frame
- Two default strategies
- `gc.collect()` before warm-up and measured timing
- DataFrame construction outside timed replay runs
- Serialization measured after `HistoricalReplayService.run()` returns
- 1 warm-up and 3 measured runs per scenario

The structure workload statistics match v0.43.0, so the workload shape is directly comparable. Wall-clock medians still remain local-machine measurements and were visibly sensitive to machine load across v0.44.0 runs.

## Final Review Conclusions

Measured facts:

- Default replay calls structure precompute once.
- Default replay calls public structure `analyze()` zero times.
- Full-frame pivot high detection runs once.
- Full-frame pivot low detection runs once.
- ATR series construction runs once.
- Optimized replay output matches prefix-reference replay output exactly in tests.
- Structure workload is unchanged from v0.43.0.
- 500-bar touch observation scans 2,715,016 cumulative rows across 8,375 zone observer calls.

Benchmark inference:

- The v0.43.0 structure cost was not primarily caused by pivot detection or ATR.
- The remaining dominant structure cost is touch observation, followed distantly by residual model/orchestration work and clustering/zone construction.
- v0.44.0 is best described as a correctness-preserving infrastructure optimization and measurement foundation, not a major performance breakthrough.

Unverified assumptions:

- Real market workloads may have different pivot, zone, and touch growth.
- A narrower touch observation optimization may provide a better risk/reward ratio than rewriting clustering, zones, and touches together.

## Remaining Bottleneck

After v0.44.0, the largest measured 500-bar segment remains structure precompute at 16173.942 ms in the review run. Inside it, touch observation dominates at 15486.017 ms. Observation construction is the next largest top-level segment at 5715.785 ms.

## Next Stage

Recommended next stage: `v0.45.0 Structure Touch Observation Optimization`.

The next stage should target touch observation first, because final review attribution shows it is the dominant measured structure subsegment. It should prove exact parity for:

- Independent touch counting
- Consecutive-inside candle suppression
- First/last observed timestamps
- Zone ordering
- Full-frame position behavior
- Future-data isolation
- Serialization equality

A broader incremental clustering/zones/touches rewrite should wait unless touch-only optimization cannot preserve parity or leaves clustering as the next dominant structure subsegment.
