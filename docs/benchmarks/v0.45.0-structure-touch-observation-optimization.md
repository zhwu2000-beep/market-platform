# v0.45.0 Structure Touch Observation Optimization

## Purpose

v0.45.0 optimizes the fully-default price-structure replay precompute path by replacing repeated per-position, per-zone prefix touch scans with per-run deduplicated point-in-time touch state keyed by exact zone bounds.

The stage preserves public touch semantics, public `observe_price_zone()` and `PriceStructureService.analyze()` APIs, structure/replay models, serialization, clustering, and injected fallback behavior.

## Existing Touch Semantics

A candle touches a zone when its high-low range intersects the inclusive zone bounds:

```text
low <= zone.upper_bound and high >= zone.lower_bound
```

Preserved semantics:

- Boundary equality counts as a touch.
- Only high and low participate; close is irrelevant.
- The first row counts if it is inside the zone.
- Consecutive inside candles count as one independent touch.
- Leaving and re-entering creates a new touch.
- `first_observed_at` and `last_observed_at` are entry-candle timestamps.
- `last_observed_at` is not the final candle of a still-open inside run.
- Count zero returns `None` for both timestamps.
- The observer scans the whole current prefix.

## Historical-Before-Availability Semantics

The optimized path intentionally keeps the existing behavior where a newly available zone can immediately report historical touches from before `zone.available_at`. Touch state is not truncated by availability time. For position `i`, the optimized observation is equivalent to observing `normalized.iloc[: i + 1]` for the current zone geometry.

## Design

`src/market_platform/structure/precompute.py` now keeps a local dictionary inside each `precompute_price_structure_snapshots()` call:

```python
dict[tuple[float, float], _TouchObservationSeries]
```

The exact touch key is:

```python
(zone.lower_bound, zone.upper_bound)
```

No rounding, tolerance, candidate identity, source method, midpoint, availability timestamp, support/resistance classification, or previous observation result is included. Equal bounds safely reuse touch state even when other zone metadata differs; near-but-not-equal bounds build separate state.

Each `_TouchObservationSeries` stores only:

- cumulative touch counts by full-frame position
- last entry index by full-frame position
- first entry index
- NumPy array payload bytes

The full-frame vector build computes:

```text
inside[i] = low[i] <= upper and high[i] >= lower
entry[0] = inside[0]
entry[i] = inside[i] and not inside[i - 1]
counts = cumulative_sum(entry)
last_entry_indices = prefix latest entry index
```

Every observed zone still constructs a fresh `ObservedPriceZone` with the current zone model and a point-in-time `PriceZoneObservation` read at the current full-frame position.

## Default/Injected Boundary

The optimization is only reachable through the existing v0.44.0 default replay gate:

- `type(service) is PriceStructureService`
- default swing-high detector identity
- default swing-low detector identity
- default ATR calculator identity
- default clusterer identity
- default observer identity

Subclassed or injected structure services still fall back to per-step public `analyze(...)`; custom observers are not adapted to vectorized state.

## Parity Methodology

Verified facts:

- Fixed-zone vector state matches public prefix `observe_price_zone(...)` at every position.
- Boundary, consecutive-inside, exit/re-entry, first/last timestamp, and no-touch semantics are preserved.
- Late-available zones include historical prefix touches before `available_at`.
- 50/100/300/500 bar precomputed snapshots match public prefix `PriceStructureService().analyze(...)` by dataclass equality, reporting dict, and JSON.
- Replay `to_dict()` parity against a public-prefix reference service is preserved.
- Future data does not change prior snapshots or replay steps.

## Review Correctness Notes

Final review added focused tests for future first-touch isolation, invalid private lookup positions, read-only stored NumPy arrays, and per-run state isolation. A future first touch does not expose `first_observed_at` or `last_observed_at` to earlier positions because `observation_at(position)` first reads `counts[position]` and returns the zero-touch model when the prefix count is zero.

The stored array payload includes only `counts` and `last_entry_indices`. Temporary `inside`, `previous_inside`, `entries`, and `entry_indices` arrays are build-time intermediates and are not counted as retained payload. Python dictionary and dataclass object overhead are also not included.
## Remaining Bottleneck

After touch optimization, observation construction is the largest measured top-level segment on this workload:

- 500-bar observation construction: 3926.918 ms
- 500-bar structure precompute: 355.893 ms

Within structure precompute, clustering/zone construction and residual model orchestration are now larger than touch build/lookup.

## Measurement Limitations

Verified facts are exact parity and workload counters from the instrumented run. Performance comparisons use the published v0.44.0 final review baseline because this implementation run did not create a separate temporary v0.44.0 checkout. The v0.44.0 report already showed local machine-load variance, so these numbers should be read as same-method local evidence rather than durable throughput promises.

## Next Stage Recommendation

Do not expand v0.45.0 into a dynamic structure state machine. The next likely bottleneck is historical observation construction and provenance/fingerprint work. Recommended next stage: v0.46.0 Historical Observation Construction Attribution. It should first break down prefix price facts extraction, signal fact conversion, structure fact conversion, provider/interval metadata, input fingerprint/provenance hashing, observation model construction, and repeated DataFrame serialization/canonicalization before choosing a narrower optimization.
